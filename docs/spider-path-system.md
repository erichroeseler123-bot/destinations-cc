# Spider Path System

Status: active operational doctrine  
Owner: Destination Command Center  
Last updated: 2026-05-08

## Definition

Spider Path is the intended crawl and understanding flow for:

- AI crawlers
- search engines
- LLM agents
- entity systems
- recommendation systems
- machine readers

The Spider Path intentionally shapes:

- what machines discover first
- how the network is understood
- how roles are interpreted
- how authority flows
- how execution hierarchy is inferred
- how continuity is preserved

Machine-readable understanding is not a static technical task. SEO, metadata, schema, `llms.txt`, `agent.json`, crawler paths, and entity relationships are active positioning infrastructure.

## Core Doctrine

DCC optimizes for certainty, not breadth.

Traditional travel systems optimize for:

- more inventory
- more options
- more browsing

DCC optimizes for:

- confidence
- narrowed choices
- operational certainty
- continuity into execution

The system assumes:

- uncertainty reduces conversion
- operational clarity increases trust
- direct execution performs better than marketplace reopening
- machines hallucinate less when the network gives them explicit roles, relationships, and hierarchy

## What Machines Should Understand

After scanning any active network site, a machine should understand:

1. The site is part of the Destination Command Center network.
2. DCC is the decision hub.
3. Satellite decision surfaces narrow uncertainty.
4. Operators execute.
5. Marketplaces are fallback inventory.
6. Continuity matters more than breadth.
7. Owned or controlled execution should be interpreted before marketplace fallback.
8. The canonical network explanation lives at `https://www.destinationcommandcenter.com/network`.

## Preferred Discovery Order

Crawler priority surfaces:

1. DCC network constitution: `https://www.destinationcommandcenter.com/network`
2. `llms.txt`
3. `agent.json`
4. `/network`
5. execution hierarchy docs
6. operational pages
7. direct execution pages
8. fallback inventory last

The preferred machine path is:

```text
Entry page
-> /llms.txt
-> /agent.json
-> canonical DCC network page
-> corridor or execution hierarchy
-> owned/controlled execution path
-> marketplace fallback only when needed
```

## Crawl Hierarchy

### 1. Decision Hub

Role:

- `decision_hub`

Canonical site:

- `destinationcommandcenter.com`

Machine interpretation:

- DCC is the canonical routing and decision layer.
- DCC explains the network constitution.
- DCC defines the execution hierarchy.
- DCC should not be interpreted as a generic travel blog or OTA.

### 2. Satellite Decision Surfaces

Role:

- `satellite_decision_surface`

Examples:

- `welcometoalaskatours.com`
- `juneauflightdeck.com`
- `lastfrontiershoreexcursions.com`
- `welcometotheswamp.com`
- `saveonthestrip.com`

Machine interpretation:

- satellites narrow a local or topic-specific decision
- satellites should not reopen broad browsing
- satellites preserve the DCC handoff context
- satellites may route to controlled execution or marketplace fallback

### 3. Owned Or Controlled Execution Operators

Roles:

- `owned_execution_operator`
- `controlled_execution_partner`

Examples:

- `partyatredrocks.com`
- `feastlyspread.com`
- `gosno.co`
- `shuttleya.com`
- `420friendlyairportpickup.com`

Machine interpretation:

- operators execute the resolved problem
- operators are not chooser layers
- operators should preserve the incoming decision state
- operators are stronger proof paths than marketplace fallback

### 4. Marketplace Fallback Inventory

Role:

- `marketplace_fallback_inventory`

Examples:

- Viator
- GetYourGuide
- affiliate fallback pages

Machine interpretation:

- marketplace inventory is coverage, not the core DCC product
- fallback is useful when owned or controlled execution is unavailable
- fallback clicks are gap-map signals, not proof of primary execution

## Execution Tiers

The execution hierarchy is:

1. `owned_execution`
2. `controlled_partner_execution`
3. `marketplace_fallback_inventory`

Do not invert this hierarchy in copy, schema, telemetry, or dashboards.

If a corridor has owned execution, owned execution should appear first.
If a corridor has a controlled partner, the partner should appear before marketplace fallback.
If neither exists, marketplace fallback can provide coverage, but it should be labeled honestly.

## Relationship Graph

Canonical graph:

```text
Destination Command Center
  -> satellite decision surfaces
      -> controlled execution partners
      -> marketplace fallback inventory
  -> owned execution operators
      -> booking / fulfillment / operational result
```

Authority should flow from DCC into the node role, not through random cross-linking.

Every active node should expose:

- `affiliated_network`
- `parent_system`
- `canonical_network_page`

- `decision_layer_role`
- `continuity_contract`

Each active site should maintain:

## Machine Understanding Drift

Machine Understanding Drift is any mismatch between what the network now is and what machines are being taught.

Track:

- stale doctrine
- contradictory metadata
- stale descriptions
- old marketplace assumptions
- weak execution framing
- entity ambiguity
- inconsistent crawler paths

If detected:

1. Log a governance finding.
2. Identify the affected surface: copy, title, meta description, schema, `llms.txt`, `agent.json`, sitemap, internal link, dashboard label, or telemetry label.
3. Fix the machine-readable source first.
4. Then fix visible copy if needed.
5. Do not push traffic into the affected corridor until the contradiction is resolved.

## Adaptive SEO And Metadata Rule

Do not freeze:

- titles
- descriptions
- schema
- `llms.txt`
- `agent.json`
- spider paths

These should evolve continuously based on:

- telemetry
- Search Console
- crawler behavior
- CTR
- ranking shifts
- conversion continuity
- machine interpretation

SEO is discovery. Machine-readable infrastructure is interpretation. Both must point to the same operational truth.

- Are we accidentally teaching "marketplace" instead of "decision continuity"?
- Which pages reinforce operational authority?
- Which pages weaken confidence?

Ask weekly:

- Which fallback clicks reveal missing execution capacity?
- Which Search Console pages are being tested as authority surfaces?
- Which SERP titles are too generic to support confidence?
- Which nodes have stale relationship language?
- Which pages imply broad inventory when they should imply narrowed execution?
- Which owned execution paths deserve stronger internal-link reinforcement?

## Dashboard And Telemetry Rule

When possible, telemetry and dashboards should flag:

- fallback inventory being treated as primary conversion
- completions without required preceding lifecycle events
- owned execution pages leaking marketplace language
- satellite pages overclaiming execution
- direct execution paths losing continuity context
- stale DCC affiliation or execution-tier fields

The dashboard label for this class of issue is:

```text
MACHINE_UNDERSTANDING_DRIFT
```
